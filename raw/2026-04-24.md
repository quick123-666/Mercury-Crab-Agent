**本轮新增工作：llm_wikify_bridge.py 语法修复**

**问题：** `ingest` 命令用旧版 `--source --type --self-create` flag 语法，导致 `llmwikify` CLI 报错 unknown option。

**根因：** llmwikify v0.30+ 将 `--source` 改为 positional argument，不再是 flag。

**修复内容（3处）：**

| 位置 | 旧语法 | 新语法 |
|------|--------|--------|
| `write_learning()` | `ingest --source <f> --type <t> --self-create` | `ingest <file>` |
| `ingest_batch()` | `ingest --source <p> --type <t>` | `ingest <path>` |
| CLI argparse | `add_argument("--source", ...)` | positional `add_argument("file_or_dir", ...)` |

**验证结果：** ✅ 修复后写入测试通过（`Bridge脚本修复测试` 成功写入 wiki，`already_exists: true` 说明识别到同名页）

**新增 corrections.md 条目（第5条）：**
> llmwikify `ingest` 命令不需要 `--self-create` flag，知识库写入直接用 positional path 即可

**经验教训：**
- `--self-create` 会触发 LLM API 调用（60s 超时），纯写入不需要此 flag
- 同一 title 写入会识别为已存在（`already_exists: true`），不会重复创建
- llmwikify positional argument 语法：`ingest <file_or_dir>`

| 问题 | 根因 | 解决 |
|------|------|------|
| `FAIL after 41.8s` 误报超时 | `print(content)` 时 GBK 无法编码 emoji（📊💡🧠），进程崩溃 | Python 脚本开头替换 `sys.stdout` 为 UTF-8 writer |
| 进程返回码非0 | emoji 写入 GBK 控制台触发 `UnicodeEncodeError` | 同上 |

**根本原因确认：**
- Windows cmd.exe 默认使用 GBK（code page 936）
- LLM 返回内容含 emoji（如 "💡 关键经验"）
- `print()` 尝试写入 GBK 编码的 stdout → 崩溃
- **不是 LLM API 超时，是 print() 失败**

**修复方案（写入脚本开头）：**
```python
import sys
sys.stdout = open(sys.stdout.fileno(), 'w', encoding='utf-8', closefd=False)
```
